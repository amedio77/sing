## 1. Executive Summary

악보 입문자가 반복적으로 막히는 3개 지점 — ① 음자리표별 계이름 위치, ② 계이름(도레미) ↔ 영어 음이름(CDE) 매칭, ③ 조표(#/♭) 붙는 순서 — 를 **짧고 반복 가능한 퀴즈 게임 3종(모드 A/B/C)** 으로 학습시키는 웹 게임을 개발한다.

핵심 가치는 "짧게, 자주, 소리와 함께"다. 정답·오답 모두 Web Audio로 실제 음을 들려주어 시각-청각을 함께 강화하고, 즉각 피드백·연속 정답(streak)·한/영 토글로 학습 루프를 완성한다.

기술적으로는 **빌드 도구·프레임워크 없이 바닐라 ES modules + SVG + Web Audio 합성**만 사용한다. 외부 의존성이 0개이므로 **Vercel에 소스 파일을 그대로 올리면(무빌드) 배포가 끝난다**. 구조는 프레임워크 없이 **플랫(파일 ~12개·폴더 2개)**: 단일 `state` 객체 + 해시 라우터가 현재 뷰를 직접 렌더하고, 3개 모드는 `src/modes.js`의 평범한 배열로 관리한다. 계층·레지스트리·pub/sub 같은 추상화는 두지 않는다(YAGNI).

개발은 **반복적 · MVP 우선** 원칙을 따른다. M0(셋업)에서 실행 가능한 뼈대를 세우고, 데이터/렌더러 공통 기반(M1)을 먼저 다진 뒤 모드 A→B→C를 순차 완성(M2~M4)한다. 각 모드는 그 자체로 플레이 가능한 세로 슬라이스(vertical slice)로 완결시켜, 언제 중단해도 동작하는 산출물을 유지한다. 이후 오디오·피드백 폴리시(M5)와 다국어·접근성·배포(M6)로 마무리한다.

**MVP 정의**: 3개 주제 각각 객관식/선택형 퀴즈 1종 + 정답음 + 즉시 피드백 + streak + 한/영 토글 + 사운드 토글 + 세션 결과 요약(정답률·문항 수). 역방향 클릭 배치, 오답 재출제, 타이머 변형·카드 매칭·전체 배치 등 확장 게임플레이, 진행 저장(영속)·별 해금, 통계/랭킹, 덧줄 확장 등은 백로그로 미룬다.

## 2. 개발 접근 / 원칙

### 2.1 핵심 원칙

| 원칙 | 이 프로젝트에서의 적용 |
|---|---|
| **반복적(Iterative)** | 각 모드를 독립적으로 완성 가능한 세로 슬라이스로 개발. 매 마일스톤 종료 시 브라우저에서 실제로 플레이 가능. |
| **MVP 우선(YAGNI)** | Must 기능만으로 학습 루프를 먼저 완성. Should/Could는 백로그로 분리해 데이터 확인 후 판단. 진행 저장·별 해금은 MVP에서 제외. |
| **KISS / DRY** | 상태 객체 1개 + 퀴즈 엔진 1개를 3모드가 공유. 새 콘텐츠 대신 기존 요소(타이머·보기 개수·범위) 조합으로 난이도 생성. |
| **단일 진실 소스** | 음이름·MIDI·조표 순서 등 음악 사실은 `data/`에 순수 상수로 1:1 반영. 로직·렌더는 이를 참조만. |
| **Fail Fast / Fail-safe** | 오디오 미지원 시 무음 폴백. 에러는 즉시 감지하되 학습 흐름은 끊지 않음. |
| **접근성 우선** | 색만으로 정답 구분 금지(색+아이콘+텍스트). 전 기능 키보드 조작. 오디오는 시각 피드백으로 대체 가능. |

### 2.2 기술 스택 및 무빌드 결정

| 항목 | 결정 | 근거 |
|---|---|---|
| 언어/모듈 | 바닐라 ES modules (`<script type="module">`, import/export) | 번들러·트랜스파일 없음 = 무빌드 만족 |
| 렌더링 | SVG (`createElementNS`) + DOM | 오선/음표 스케일링, 이벤트 바인딩 용이 |
| 오디오 | Web Audio API (`OscillatorNode` triangle/sine + `GainNode`) | 음원 파일 불필요, `midiToFreq` 공식 하나로 통일 |
| 상태 | 단일 `state` 객체 + 해시 라우터 직접 렌더 | 프레임워크·구독 불필요, `setState(patch)` → 현재 뷰 1회 재렌더 |
| 저장 | (후순위) localStorage 래퍼 | MVP 제외 — 백로그. 필요 시 `src/storage.js` 추가 |
| **실행/배포** | **Vercel 정적 배포(무빌드)** + 로컬은 `npx serve` / `vercel dev` | HTTPS 서빙이라 ES modules 정상. §8 참조 |

### 2.3 아키텍처 (플랫 구조)

계층·레지스트리·pub/sub 없이 **플랫 파일 구성**을 채택한다.

```
/          index.html, styles.css
data/      notes.js, clefs.js, keys.js      (순수 상수)
src/       audio.js, staff.js, i18n.js, game.js, modes.js, ui.js, main.js
           (storage.js는 후순위/선택)
```

- **상태 관리**: 단일 `state` 객체. `main.js`의 해시 라우터가 현재 뷰의 `render(state)`를 **직접 호출**한다. `setState(patch)`는 상태를 갱신한 뒤 현재 뷰를 1회 재렌더한다. 구독자 Set·subscribe/unsubscribe 없음.
- **모드**: 3개 모드는 `src/modes.js`의 배열 `[{id, name, generate, render, check}, ...]`. 레지스트리·플러그인·전략 패턴 없음. 모드 인터페이스는 주석 한 줄로 표기.
- **병합**: `state.js`+`quiz.js` → `game.js` / `menu.js`+`result.js`+`components.js` → `ui.js` / `lib`·`core` 폴더 삭제.
- **유지(적정)**: `i18n.js` 딕셔너리, (후순위) `storage.js` 래퍼는 현행 유지.

## 4. 마일스톤 로드맵

```
M0 셋업/스캐폴딩
      │
      ▼
M1 오선 데이터 & 렌더러 (공통 기반: data/, src/)
      │
      ├──────────────┬──────────────┐
      ▼              ▼              ▼
   M2 모드 A       M3 모드 B      M4 모드 C
 (음 위치 읽기)  (계이름↔영어)   (조표 순서)
      └──────────────┴──────────────┘
                     │
                     ▼
        M5 오디오 & 피드백 폴리시
                     │
                     ▼
   M6 다국어 · 접근성 · 반응형 · 배포
```

| 마일스톤 | 목표 | 핵심 산출물 | 성격 |
|---|---|---|---|
| **M0** | 실행 가능한 앱 셸 | 라우터·`state`·빈 메뉴 | 셋업 |
| **M1** | 3모드가 공유할 기반 완성 | data/·오선 렌더러·오디오·퀴즈 엔진·모드 배열·i18n | 기반(가장 중요) |
| **M2** | 첫 완결 게임(수직 슬라이스) | 모드 A 플레이 가능 | MVP 코어 |
| **M3** | 두 번째 게임 | 모드 B 플레이 가능 | MVP 코어 |
| **M4** | 세 번째 게임 | 모드 C 플레이 가능 | MVP 코어 |
| **M5** | 학습 경험 완성도 | 오디오·피드백·streak 통합 | 폴리시 |
| **M6** | 출시 준비 | 한/영·접근성·반응형·Vercel 배포 URL | 릴리스 |

> M2~M4는 M1 완료 후 병렬화 가능하나, 1인 개발 기준으로는 **A→B→C 순차**를 권장한다(홈 화면의 추천 학습 순서와 일치, 모드 A에서 확립한 패턴을 B·C가 재사용). 모드 A가 오선 렌더러를 가장 많이 사용하므로 먼저 완성해 렌더러를 검증한다.

## 6. 테스트 전략

프레임워크·빌드가 없으므로 **수동 QA 체크리스트 중심**으로 운영한다. 단, 음악 사실 데이터와 주파수 공식만은 콘솔에서 실행하는 **경량 단정(assert) 스니펫**으로 자동 검증한다(회귀 방지, 무빌드 유지).

### 6.1 테스트 계층

| 계층 | 방식 | 대상 |
|---|---|---|
| 데이터/공식 검증 | 브라우저 콘솔 assert 스니펫(무빌드) | `midiToFreq`, 음↔MIDI 매핑, 조표 순서 배열 |
| 모드별 기능 QA | 수동 체크리스트 | 문제 생성·판정·피드백·오디오 |
| 통합 QA | 수동 시나리오 | 메뉴→모드→결과→재도전 흐름, 언어/사운드 토글 |
| 접근성/호환 QA | 수동 + 브라우저 도구 | 키보드·대비·색약·반응형·크로스 브라우저 |

### 6.2 음악 이론 정확성 검증 (필수 · 최우선)

학습 게임의 신뢰도는 음악 사실의 정확성에 달려 있다. 아래는 **표준 사실과 1:1 대조**하는 필수 항목이다.

| 검증 항목 | 기대값 (표준) | 확인 방법 |
|---|---|---|
| 주파수 공식 | `midiToFreq(60)≈261.63`, `(69)=440`, `(72)≈523.25` | 콘솔 assert |
| Treble 줄(아래→위) | 미·솔·시·레·파 / E G B D F | 오선 렌더 육안 + 데이터 대조 |
| Treble 칸(아래→위) | 파·라·도·미 / F A C E (FACE) | 동일 |
| Bass 줄(아래→위) | 솔·시·레·파·라 / G B D F A | 동일 |
| Bass 칸(아래→위) | 라·도·미·솔 / A C E G | 동일 |
| Treble 기준음 | 둘째 줄 = G4 | `noteToY` 앵커 확인 |
| Bass 기준음 | 넷째 줄 = F3 | `noteToY` 앵커 확인 |
| 공유음 처리 | Middle C = C4가 두 자리표에서 일관 | 덧줄 렌더 대조 |
| 샤프 순서 | 파·도·솔·레·라·미·시 / F C G D A E B | 데이터 assert |
| 플랫 순서 | 시·미·라·레·솔·도·파 / B E A D G C F (샤프 정확한 역순) | `FLAT_ORDER === reverse(SHARP_ORDER)` assert |
| 계이름↔영어 | 도C 레D 미E 파F 솔G 라A 시B (고정도) | 데이터 assert |
| 정답음 일치 | 화면 음표의 이름 = 재생되는 주파수의 음 | 청음 + 데이터 교차 확인 |

> 검증 원칙: 렌더된 음표 위치·정답 판정·재생 음이 **세 곳 모두 동일 사실**을 가리켜야 한다(위치=이름=소리 삼중 일치). 하나라도 어긋나면 학습에 해로우므로 릴리스 차단(blocker)으로 취급.

### 6.3 모드별 수동 QA 체크리스트 (발췌)

**모드 A**
- [ ] 임의 음표가 오선의 올바른 줄/칸에 표시된다.
- [ ] 4지선다 중 정답 선택 시 초록+아이콘+"정답!" + 해당 음 재생.
- [ ] 오답 시 정답 보기·정답 위치가 강조되고 정답음·니모닉이 나온다.
- [ ] Lv1(treble)→Lv3(bass)→Lv4(혼합) 전환 시 자리표 아이콘이 항상 보인다.
- [ ] 덧줄 음(Lv5, 선택)이 있으면 덧줄이 그려진다.

**모드 B**
- [ ] "도" 제시 → "C" 정답 처리(양방향 모두).
- [ ] 언어 토글 시 프롬프트·보기 표기가 도레미↔CDE로 즉시 바뀐다.
- [ ] 보기/카드 상호작용마다 해당 음이 재생된다.
- [ ] 타이머 Lv에서 제한시간 바가 줄고 시간 초과가 처리된다.

**모드 C**
- [ ] 샤프 "다음 하나"가 파→도→솔… 순서로만 정답 처리된다.
- [ ] 플랫 진입 시 역순 니모닉이 강조된다.
- [ ] "샤프 3번째는?"에 '솔' 정답 처리.

**공통/통합**
- [ ] 메뉴→모드→결과→[다시하기]/[홈] 흐름이 끊기지 않는다.
- [ ] 사운드 Off 시 소리가 나지 않지만 시각 피드백은 유지된다.
- [ ] streak가 연속 정답에 증가하고 오답 시 리셋된다.
- [ ] 결과 화면에 세션 정확도%·세션 내 최고 streak·문제 수가 표시된다.

### 6.4 접근성 · 호환 QA
- [ ] Tab/Enter/화살표만으로 전 게임 완주.
- [ ] 색약 시뮬레이터에서 정답/오답 구분 가능(색 외 단서 존재).
- [ ] 명도 대비 WCAG AA 통과(브라우저 접근성 도구).
- [ ] 터치 타깃 ≥48×48px(모바일 실기기 또는 디바이스 에뮬레이션).
- [ ] **가로(landscape)·세로(portrait) 전환 시 레이아웃 정상**(가로=좌우 2단, 세로=상하 1열).
- [ ] **320px 소형 화면**에서 조표 7칩이 2행 wrap 또는 반응형 축소로 잘림 없이 표시되고 터치 하한(48px)을 유지한다.
- [ ] **설정(⚙) 화면**에서 한/영 토글·사운드 On/Off가 정상 동작한다.
- [ ] Chrome·Edge·Safari·Firefox 최신 버전에서 스모크 통과.
- [ ] 오디오 미지원 브라우저에서 무음 폴백 + 기능 정상.

## 8. 배포 (Vercel 정적 배포)

### 8.1 왜 Vercel 정적 배포인가
빌드 산출물이 없고 외부 의존성이 0이므로, **소스 파일을 그대로 정적 호스팅**하면 된다. Vercel은 무빌드 정적 사이트를 HTTPS로 서빙하며 GitHub 연동 자동배포를 제공하여 이 프로젝트에 최적이다. HTTPS 서빙이므로 ES modules(`<script type="module">`)가 별도 설정 없이 정상 동작한다.

### 8.2 Vercel 정적 배포 방법 (구체 절차)

| 항목 | 설정 |
|---|---|
| **Framework Preset** | **Other** (프레임워크 감지 불필요) |
| **Build Command** | **없음(비움)** — 무빌드 |
| **Output Directory** | **루트(`.`)** — 소스가 곧 배포물 |
| **Install Command** | 없음 (의존성 0) |
| **vercel.json** | **불필요(YAGNI)** — 정적 파일은 기본 설정으로 서빙됨 |

**배포 트리거 (택1)**
1. **GitHub 연결 + push 자동배포(권장)**: 리포지토리를 Vercel에 Import → 위 설정 확인 → 이후 `main`에 push할 때마다 자동 배포. PR별 프리뷰 URL 자동 생성.
2. **CLI**: 프로젝트 루트에서 `vercel --prod` 실행. (초기 1회 `vercel` 로 프로젝트 링크.)

**로컬 개발**
- `npx serve` 또는 `vercel dev` — 정적 서버 한 줄로 실행. 별도 빌드 없음.
- 로컬도 `http://localhost`(HTTP) 서빙이므로 ES modules 정상 동작.

### 8.3 배포 체크리스트
- [ ] 모든 자원이 상대 경로이며 외부 요청(CDN·원격 폰트)이 0회다(네트워크 탭 확인).
- [ ] Vercel 배포 URL(HTTPS)에서 ES modules가 정상 로드된다.
- [ ] AudioContext가 배포 환경에서 제스처 후 활성화된다.
- [ ] push(또는 `vercel --prod`) 시 자동 배포가 최신 커밋을 반영한다.
- [ ] 4개 브라우저·모바일 실기기에서 스모크 통과.
- [ ] README에 실행 방법(`npx serve`/`vercel dev`)과 Vercel 배포 절차가 문서화되어 있다.

> **성공지표(MVP 스코프)**: MVP는 **세션 단위 측정만** 한다 — 세션 정답률, 세션 내 최고 streak, 세션 문제 수. 세션 간 영속 저장이 필요한 지표(누적 추세·최고 기록)는 진행 저장(백로그)이 들어온 뒤로 명시적으로 미룬다.
